# Warm-Flow 动态表单与流程编排可视化验证设计

## 1. 背景与目标

本次工作不是直接建设完整业务系统，而是基于当前仓库已有的 Warm-Flow 能力，搭建一个**最小、可手动操作、可观察真实行为**的验证场景，帮助快速建立对以下能力的直观认知：

1. 流程引擎的流程编排与条件分支能力
2. `formCustom = Y` 动态表单的配置、绑定、提交流程
3. 动态表单字段参与流程分支判断时的真实行为
4. 不同节点绑定不同动态表单时，页面渲染与数据透传的表现
5. 将 `formCreate` 页面通过 iframe 方式挂到宿主页后，父子页面的初始化与提交交互

本设计明确覆盖两条验证线：

- **框架内闭环验证**
  - 使用 Warm-Flow 自带的动态表单、流程设计器、表单填报页完成全链路验证
- **外挂宿主页接入验证**
  - 使用一个最小 iframe 宿主页承载 `formCreate`，模拟未来接入 IRA 前端时的交互方式

## 2. 设计范围

### 2.1 本次要做

本次只做一个最小但完整的请假流程验证场景：

- 发起节点绑定动态表单 A
- 条件判断节点根据 `leaveDays` 分支
- 短假审批节点绑定动态表单 B
- 长假审批节点绑定动态表单 C
- 结束节点收口流程

同时补一个最小宿主页，用于：

- iframe 打开 `type=formCreate`
- 响应 `formInit`
- 传递 `taskId`、`type`、`disabled` 等初始化参数
- 接收 `submitSuccess`
- 手动切换“申请人 / 主管 / 经理”视角

### 2.2 本次不做

本次不做以下扩展工作：

- 不接真实 IRA 前端工程
- 不引入完整登录、权限系统与真实身份体系
- 不重构 Warm-Flow 运行时变量模型
- 不做复杂的条件表达式编辑器
- 不做生产可用的业务页面与业务数据库落库方案
- 不实现复杂的并行、会签、减签、转办等高级审批场景

## 3. 场景定义

### 3.1 流程拓扑

第一版测试流程固定为条件分支型 `B` 结构：

```text
发起申请
  -> 条件判断（leaveDays）
    -> leaveDays <= 3 -> 主管审批
    -> leaveDays > 3  -> 经理审批
  -> 结束
```

### 3.2 业务语义

采用“请假申请”作为演示场景，只保留一个核心分支字段，避免噪音过多。

- 关键字段：`leaveDays`
- 目标验证：
  - `leaveDays = 2` 时走主管审批
  - `leaveDays = 5` 时走经理审批

### 3.3 节点表单分配

每个关键节点绑定一份独立动态表单，用于观察“历史只读、当前可写、未来隐藏”的渲染行为。

#### 发起节点：动态表单 A

- `leaveDays`：请假天数，数值型
- `reason`：请假原因，多行文本
- `applicantRemark`：申请补充说明，文本

#### 主管审批节点：动态表单 B

- `leaderOpinion`：主管意见，文本
- `leaderExtraNote`：主管额外补充，文本

#### 经理审批节点：动态表单 C

- `managerOpinion`：经理意见，文本
- `managerExtraNote`：经理额外补充，文本

> 说明  
> 审批节点特意增加了“意见 + 自定义输入框”两个字段，不是为了模拟最终业务，而是为了更清楚地验证不同节点的动态表单是否真的独立渲染、独立提交。

## 4. 关键技术假设

### 4.1 动态表单绑定语义

本次验证基于当前仓库已明确的真实约定：

- 使用 Warm-Flow 自带动态表单能力配置表单
- 节点使用 `formCustom = Y`
- `formPath` 存储动态表单 ID
- 运行时表单值进入流程变量 `formData`

### 4.2 条件分支变量引用

条件表达式按当前仓库已收口的约定使用：

```text
formData.<field>
```

本次核心表达式示例：

```text
${formData.leaveDays > 3}
```

对应反向分支可使用：

```text
${formData.leaveDays <= 3}
```

### 4.3 宿主页交互契约

外挂宿主页不直接参与流程引擎运算，只承担容器与桥接职责：

- 承载 `type=formCreate` 的 iframe
- 监听子页发出的 `formInit`
- 向子页回传初始化参数
- 监听 `submitSuccess`
- 手动刷新任务视图或切换操作者视角

## 5. 验证设计

## 5.1 第一阶段：框架内闭环验证

### 目标

证明 Warm-Flow 自带能力内部可以独立完成以下链路：

```text
动态表单配置
  -> 流程节点绑定
  -> 表单填写并提交
  -> 变量进入 formData
  -> 网关 / skip 条件分支
  -> 后续节点表单渲染
```

### 验证步骤

1. 配置三份动态表单 A / B / C
2. 配置一份带条件分支的请假流程
3. 将 A 绑定到发起节点，将 B 绑定到主管节点，将 C 绑定到经理节点
4. 先以同一个测试用户跑通整条链路
5. 先提交 `leaveDays = 2`，验证走主管分支
6. 再提交 `leaveDays = 5`，验证走经理分支

### 第一阶段用户策略

为降低调试复杂度，第一阶段允许使用**同一个测试用户**覆盖全部节点。

这样做的目的：

- 先验证流程与表单链路本身成立
- 不让“身份切换”掩盖“变量或分支问题”

这不是最终用户模型，而是第一版认知验证策略。

### 第一阶段成功标准

满足以下条件即可视为第一阶段达标：

1. 三份动态表单都能成功配置并绑定到对应节点
2. `leaveDays = 2` 时进入主管审批节点
3. `leaveDays = 5` 时进入经理审批节点
4. 主管审批页展示：
   - 发起表单只读
   - 主管表单可写
   - 经理表单不展示
5. 经理审批页展示：
   - 发起表单只读
   - 经理表单可写
   - 主管表单不展示
6. 审批节点的意见字段和额外输入字段都可填写并提交成功

## 5.2 第二阶段：外挂宿主页接入验证

### 目标

证明 `formCreate` 页面可以通过最小宿主页方式被外部系统承载，并建立对未来 IRA 前端接入形态的直观认知。

### 设计原则

宿主页只做“薄壳”，不做成完整业务前端。

它只关注：

- iframe 怎么挂
- 父子页面怎么通信
- 不同用户 / 不同任务怎么切换观察

### 宿主页最小功能

宿主页固定包含四个区块：

#### 1. 用户切换区

- `申请人`
- `主管`
- `经理`

用于模拟不同节点属于不同操作者的视角切换。

#### 2. 任务入口区

- 发起流程
- 打开当前待办
- 打开历史查看

#### 3. 调试信息区

显示：

- 当前发送给 iframe 的初始化参数
- 最近一次收到的 `formInit`
- 最近一次收到的 `submitSuccess`

#### 4. iframe 展示区

真正承载：

```text
/warm-flow-ui/index.html?type=formCreate
```

### 第二阶段交互模型

```text
宿主页
  -> 等待 iframe 发起 formInit
  -> 回传 taskId / formId / type / disabled
  -> 用户在 iframe 内提交
  -> iframe 发出 submitSuccess
  -> 宿主页更新任务视图
```

### 第二阶段成功标准

满足以下条件即可视为第二阶段达标：

1. iframe 能正常打开 `formCreate`
2. 宿主页能捕获并响应 `formInit`
3. 宿主页能向子页成功回传初始化参数
4. 表单提交成功后能收到 `submitSuccess`
5. 切换“申请人 / 主管 / 经理”视角时，可以重新打开对应任务页面观察节点差异

## 6. 界面与体验预期

### 6.1 节点渲染预期

#### 发起节点

- 只显示发起表单 A
- 可编辑
- 审批动作区按发起态展示

#### 主管审批节点

- 发起表单 A：只读
- 主管表单 B：可编辑
- 经理表单 C：隐藏

#### 经理审批节点

- 发起表单 A：只读
- 主管表单 B：隐藏
- 经理表单 C：可编辑

### 6.2 视觉验证重点

本次可视化验证需要重点观察：

- 表单区块是否按节点切换正确呈现
- 历史节点数据是否真正带下来了
- 不同分支下页面是否真的切到了不同审批表单
- 外层宿主页是否能清楚展示父子页面消息往来

## 7. 风险与边界

### 7.1 主要风险

1. 条件表达式虽能保存，但运行时变量路径写错会导致流转失败
2. 同一测试用户跑全链路时，可能掩盖部分真实权限问题
3. 宿主页只是调试壳，不代表真实 IRA 前端最终结构
4. 如果当前仓库局部实现与文档存在偏差，需要以实际运行结果为准修正认知

### 7.2 风险控制方式

1. 第一阶段只保留一个分支字段 `leaveDays`
2. 先框架内闭环，再做宿主页，不把两类问题混排
3. 第二阶段只模拟用户切换，不引入真正权限系统
4. 所有结论都必须以实际操作与页面结果验证

## 8. 后续实现顺序

实现阶段推荐按以下顺序推进：

1. 建立三份动态表单定义
2. 建立请假流程定义与条件分支
3. 跑通同一测试用户下的框架内闭环
4. 建立最小宿主页并接入 `formCreate`
5. 补充“申请人 / 主管 / 经理”视角切换
6. 形成一份手动操作说明，记录如何发起、如何切换分支、如何观察渲染结果

## 9. 验收结论

本设计的目标不是交付最终业务系统，而是交付一个**最小但真实的可视化认知样板**。

只要该样板能同时回答下面两个问题，本次设计就成立：

1. 使用 Warm-Flow 自带动态表单时，字段能否驱动条件分支并正常流转？
2. 将 `formCreate` 挂到外部宿主页后，动态表单与节点渲染的整体交互形态是什么样？

如果两点都能通过实际操作获得清晰答案，就说明该样板已经达到目的。
