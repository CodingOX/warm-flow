# Warm-Flow 动态表单与流程编排可视化验证设计

## 1. 背景与目标

本次工作不是直接建设完整业务系统，而是基于当前仓库已有的 Warm-Flow 能力，搭建一个**最小、可手动操作、可观察真实行为**的验证场景，帮助快速建立对以下能力的直观认知：

1. 流程引擎的流程编排与条件分支能力
2. `formCustom = Y` 动态表单的配置、绑定、提交流程
3. 系统自带表单配置界面的能力边界与使用方式
4. 动态表单字段参与流程分支判断时的真实行为
5. 不同节点绑定不同动态表单时，页面渲染与数据透传的表现
6. 将 `formCreate` 页面通过 iframe 方式挂到宿主页后，父子页面的初始化与提交交互

本设计明确覆盖两条验证线：

- **框架内闭环验证**
  - 使用 Warm-Flow 自带的表单配置界面、流程设计器、表单填报页完成全链路验证
- **外挂宿主页接入验证**
  - 使用一个最小 iframe 宿主页承载 `formCreate`，模拟未来接入 IRA 前端时的交互方式

## 2. 设计范围

### 2.1 本次要做

本次只做一个最小但完整的请假流程验证场景，且验证从“表单配置界面”开始：

- 通过系统自带表单配置界面生成发起动态表单 A
- 条件判断节点根据 `leaveDays` 分支
- 通过系统自带表单配置界面生成短假审批动态表单 B
- 通过系统自带表单配置界面生成长假审批动态表单 C
- 结束节点收口流程

同时补一个最小宿主页，用于：

- iframe 打开 `type=formCreate`
- 响应 `formInit`
- 传递 `taskId`、`type`、`disabled` 等初始化参数
- 接收 `submitSuccess`
- 手动切换“申请人 / 主管 / 经理”视角

### 2.2 本次不做

本次不做以下扩展工作：

- 不接真实 IRA 前端工程
- 不引入完整登录、权限系统与真实身份体系
- 不重构 Warm-Flow 运行时变量模型
- 不做复杂的条件表达式编辑器
- 不做生产可用的业务页面与业务数据库落库方案
- 不实现复杂的并行、会签、减签、转办等高级审批场景

## 3. 场景定义

### 3.1 流程拓扑

第一版测试流程固定为条件分支型 `B` 结构：

```text
发起申请
  -> 条件判断（leaveDays）
    -> leaveDays <= 3 -> 主管审批
    -> leaveDays > 3  -> 经理审批
  -> 结束
```

### 3.2 业务语义

采用“请假申请”作为演示场景，只保留一个核心分支字段，避免噪音过多。

- 关键字段：`leaveDays`
- 目标验证：
  - `leaveDays = 2` 时走主管审批
  - `leaveDays = 5` 时走经理审批
  - `leaveDays` 必须出现在分支前的发起表单中

### 3.3 节点表单分配

每个关键节点绑定一份独立动态表单，用于观察“历史只读、当前可写、未来隐藏”的渲染行为。

这些动态表单不是预置页面，也不是手写业务前端，而是必须通过 **Warm-Flow 自带表单配置界面** 实际配置生成，再绑定到对应节点。

#### 发起节点：动态表单 A

- 必须包含 `leaveDays`：请假天数，正整数
- 建议再补 1 到 2 个普通字段，例如：
  - `reason`：请假原因，多行文本
  - `applicantRemark`：申请补充说明，文本

#### 主管审批节点：动态表单 B

- 至少包含 1 个意见类输入字段，例如：
  - `leaderOpinion`：主管意见，文本
- 可以再补 1 个简单自定义输入项，例如：
  - `leaderExtraNote`：主管额外补充，文本

#### 经理审批节点：动态表单 C

- 至少包含 1 个意见类输入字段，例如：
  - `managerOpinion`：经理意见，文本
- 可以再补 1 个简单自定义输入项，例如：
  - `managerExtraNote`：经理额外补充，文本

> 说明  
> 审批节点特意要求“至少一个意见字段，最好再加一个简单输入项”，不是为了模拟最终业务，而是为了更清楚地验证不同节点的动态表单是否真的独立渲染、独立提交。  
> 字段名可以按实际配置界面能力微调，但发起节点的 `leaveDays` 是硬约束。
> 第一版默认把 `leaveDays` 约束为正整数，不接受空值、负数或小数，避免把验证重点从流程分支带偏到字段边界处理。

## 4. 关键技术假设

### 4.1 动态表单绑定语义

本次验证基于当前仓库已明确的真实约定：

- 使用 Warm-Flow 自带动态表单能力配置表单
- 在当前仓库语义中，节点使用 `formCustom = Y` 表示启用系统自带动态表单配置结果，而不是页面路径模式
- `formPath` 存储动态表单 ID
- 运行时表单值进入流程变量 `formData`

与之对应：

- `formCustom = N`
  - `formPath` 表示页面路径或外部表单入口
- `formCustom = Y`
  - `formPath` 表示动态表单主键 ID
  - 节点实际使用的是系统自带表单配置界面产出的动态表单

### 4.1.1 表单配置界面纳入首轮验证范围

第一版必须从系统自带表单配置界面开始验证，而不是跳过配置过程直接假设表单已存在。

本次要确认的事实包括：

- 配置界面能创建哪些基础字段
- 表单保存后的标识如何被流程节点引用
- 配好的表单是否能被 `formCustom = Y` 节点正常绑定
- 配好的表单在运行页和宿主页中是否能按预期渲染

### 4.2 条件分支变量引用

条件表达式按当前仓库已收口的约定使用：

```text
formData.<field>
```

第一版表达式策略固定为：

```text
default@@${...}
```

不在第一版同时验证 `spel@@#{...}` 或 `snel@@#{...}`，避免把验证重点从动态表单分支链路带偏到多策略兼容。

本次核心表达式示例：

```text
default@@${formData.leaveDays > 3}
```

对应反向分支可使用：

```text
default@@${formData.leaveDays <= 3}
```

### 4.3 宿主页交互契约

外挂宿主页不直接参与流程引擎运算，只承担容器与桥接职责：

- 承载 `type=formCreate` 的 iframe
- 监听子页发出的 `formInit`
- 向子页回传初始化参数
- 监听 `submitSuccess`
- 手动刷新任务视图或切换操作者视角

### 4.3.1 通信机制

第二阶段固定使用：

```text
window.postMessage
```

不采用 URL 参数回传运行态任务上下文，也不在第一版引入额外 RPC 桥。

### 4.3.2 已确认的消息方向

#### 子页 -> 父页

初始化请求：

```json
{ "method": "formInit" }
```

提交成功通知：

```json
{ "method": "submitSuccess" }
```

高度同步：

```json
{ "method": "getOffsetHeight", "offsetHeight": 820 }
```

#### 父页 -> 子页

初始化回传：

```json
{
  "method": "formInit",
  "data": {
    "taskId": "123",
    "formId": "456",
    "type": "0",
    "disabled": false
  }
}
```

重置请求：

```json
{ "method": "reset" }
```

### 4.3.3 `formInit` 消息体约定

第一版固定使用以下 schema：

- `taskId`: string
  - `type = "0"` 或 `type = "1"` 时必填
- `formId`: string
  - `type = "2"` 时必填
- `type`: string
  - `"0"`：待办办理
  - `"1"`：已办历史
  - `"2"`：已发布表单预览
- `disabled`: boolean
  - 是否禁用表单输入

### 4.3.4 第一版错误处理约定

第一版先采用最小错误策略：

1. iframe 发出 `formInit` 后，如果父页未回传数据，则子页保持空白待初始化状态
2. 父页必须在调试区打印最近一次 `formInit` 请求和回传 payload
3. 如果 `taskId` / `formId` 缺失导致加载失败，按当前页面告警行为处理
4. `formInit` 超时不做自动重试，第一版由人工重新触发打开流程或刷新页面

## 5. 验证设计

### 5.1 第一阶段：框架内闭环验证

### 目标

证明 Warm-Flow 自带能力内部可以独立完成以下链路：

```text
表单配置界面创建动态表单
  -> 流程节点绑定
  -> 表单填写并提交
  -> 变量进入 formData
  -> 网关 / skip 条件分支
  -> 后续节点表单渲染
```

### 验证步骤

1. 打开系统自带表单配置界面，理解可配置字段与保存方式
2. 通过配置界面创建三份动态表单 A / B / C
3. 确认发起表单 A 中包含 `leaveDays`
4. 配置一份带条件分支的请假流程
5. 将 A 绑定到发起节点，将 B 绑定到主管节点，将 C 绑定到经理节点
6. 先以同一个测试用户跑通整条链路
7. 先提交 `leaveDays = 2`，验证走主管分支
8. 再提交 `leaveDays = 5`，验证走经理分支

### 第一阶段用户策略

为降低调试复杂度，第一阶段允许使用**同一个测试用户**覆盖全部节点。

这样做的目的：

- 先验证流程与表单链路本身成立
- 不让“身份切换”掩盖“变量或分支问题”

这不是最终用户模型，而是第一版认知验证策略。

第一阶段不做真实权限切换，也不强依赖不同账号登录。

第一版只要求：

- 同一个测试用户能够串行跑通所有节点
- 后续在宿主页层通过不同任务入口或预置任务上下文模拟“申请人 / 主管 / 经理”视角

### 第一阶段成功标准

满足以下条件即可视为第一阶段达标：

1. 三份动态表单都能成功配置并绑定到对应节点
2. 发起表单中的 `leaveDays` 能被成功录入并参与流程判断
3. `leaveDays = 2` 时进入主管审批节点
4. `leaveDays = 5` 时进入经理审批节点
5. 主管审批页展示：
   - 发起表单只读
   - 主管表单可写
   - 经理表单不展示
6. 经理审批页展示：
   - 发起表单只读
   - 经理表单可写
   - 主管表单不展示
7. 审批节点的意见字段和额外输入字段都可填写并提交成功

### 5.2 第二阶段：外挂宿主页接入验证

### 目标

证明 `formCreate` 页面可以通过最小宿主页方式被外部系统承载，并建立对未来 IRA 前端接入形态的直观认知。

### 设计原则

宿主页只做“薄壳”，不做成完整业务前端。

它只关注：

- iframe 怎么挂
- 父子页面怎么通信
- 不同用户 / 不同任务怎么切换观察

### 宿主页最小功能

宿主页固定包含四个区块：

#### 1. 用户切换区

- `申请人`
- `主管`
- `经理`

用于模拟不同节点属于不同操作者的视角切换。

第一版这里的“切换用户”不是要求先做完整权限系统，而是允许通过以下任一方式模拟：

- 切换不同 `taskId`
- 切换不同初始化 payload
- 切换不同宿主页入口按钮

#### 2. 任务入口区

- 发起流程
- 打开当前待办
- 打开历史查看

#### 3. 调试信息区

显示：

- 当前发送给 iframe 的初始化参数
- 最近一次收到的 `formInit`
- 最近一次收到的 `submitSuccess`

#### 4. iframe 展示区

真正承载：

```text
/warm-flow-ui/index.html?type=formCreate
```

### 第二阶段交互模型

```text
宿主页
  -> 等待 iframe 发起 formInit
  -> 回传 taskId / formId / type / disabled
  -> 用户在 iframe 内提交
  -> iframe 发出 submitSuccess
  -> 宿主页更新任务视图
```

### 第二阶段成功标准

满足以下条件即可视为第二阶段达标：

1. iframe 能正常打开 `formCreate`
2. 宿主页能捕获并响应 `formInit`
3. 宿主页能向子页成功回传初始化参数
4. 表单提交成功后能收到 `submitSuccess`
5. 切换“申请人 / 主管 / 经理”视角时，可以重新打开对应任务页面观察节点差异

## 6. 界面与体验预期

### 6.1 节点渲染预期

#### 发起节点

本次可视化验证需要重点观察：

- 表单区块是否按节点切换正确呈现
- 历史节点数据是否真正带下来了
- 不同分支下页面是否真的切到了不同审批表单
- 外层宿主页是否能清楚展示父子页面消息往来

## 7. 风险与边界

### 7.1 工程落地结构

### 7.1.1 目录划分原则

当前仓库本质上是 Warm-Flow 框架源码仓库，而不是业务应用仓库。

从根 `pom.xml` 可见，现有模块主要分为三层：

- `warm-flow-core`：流程引擎核心
- `warm-flow-orm`：ORM 适配层
- `warm-flow-plugin`：插件层

其中 UI 相关能力再细分为：

```text
warm-flow-ui
  -> warm-flow-plugin-vue3-ui
  -> warm-flow-plugin-ui-core
  -> warm-flow-plugin-ui-sb-web
```

因此，本次验证不建议把 demo 业务代码直接塞进上述框架模块中。

### 7.1.2 推荐新增模块

建议在当前仓库根目录新增一个单独的业务联调模块，例如：

```text
warm-flow-demo-app
```

它的定位是：

- 本地 Spring Boot 启动应用
- 当前验证场景的业务承载壳
- iframe 宿主页与调试页容器
- 流程发起、任务打开、用户切换等薄交互入口

### 7.1.3 推荐目录形态

```text
warm-flow
  ├─ warm-flow-core
  ├─ warm-flow-orm
  ├─ warm-flow-plugin
  ├─ warm-flow-ui
  └─ warm-flow-demo-app
       ├─ 启动类
       ├─ application.yml
       ├─ demo 初始化 SQL / 测试数据
       ├─ 宿主页页面
       ├─ iframe 调试页
       └─ demo 接口层
```

### 7.1.4 为什么不把业务直接写进框架模块

原因很明确：

1. `warm-flow-core`、`warm-flow-plugin-*` 的职责是框架能力，不应该混入 demo 业务逻辑
2. 将 demo 代码单独放在 `warm-flow-demo-app`，可以避免污染框架模块边界
3. 后续如果删掉或替换 demo 应用，不会影响框架主干结构
4. 这样更适合长期保留为“联调实验场”，同时支持反复改源码验证

### 7.1.5 模块职责约束

本次建议按以下边界落地：

- `warm-flow-core`
  - 不放业务 controller
  - 不放宿主页
  - 不放 demo SQL
- `warm-flow-plugin-ui-*`
  - 保持框架 UI 能力与接口职责
  - 仅在需要修框架行为时修改
- `warm-flow-demo-app`
  - 放本次请假流程验证壳
  - 放宿主页与 iframe 承载页
  - 放用户切换、任务调试入口

### 7.2 源码调试与测试策略

### 7.2.1 总体原则

开发环境不采用“先发线上 jar，再回到业务工程验证”的模式。

本次验证默认采用：

- **源码联调**
- **本地多模块构建**
- **改完即重启 / 重跑验证**

### 7.2.2 Java 侧联调方式

`warm-flow-demo-app` 应直接依赖当前仓库中的源码模块，而不是依赖远端已发布产物。

推荐依赖方向：

- `warm-flow-*-starter`：选择当前实验使用的 Spring Boot starter
- `warm-flow-plugin-ui-sb-web`：承载表单配置界面、流程设计器、`formCreate` 等 UI Web 能力

在同一个 Maven reactor 中构建时，Maven 会优先使用当前仓库中的源码模块产物。

这意味着开发时可以直接修改：

- 引擎流转逻辑
- 动态表单绑定逻辑
- 条件表达式相关逻辑
- UI 后端接口

然后立即在 demo 应用中验证。

### 7.2.3 前端侧联调方式

前端联调分两种情况：

#### 情况 A：只验证现有 UI 能力

优先直接使用当前仓库已有 UI 能力，不先改 `warm-flow-ui` 源码。

适用于：

- 先认识表单配置界面怎么用
- 先认识 `formCreate` 的默认行为
- 先跑通流程和宿主页链路

#### 情况 B：需要修改 UI 行为或展示

如果现有 UI 能力不足以完成验证，再进入 `warm-flow-ui` 源码层修改。

此时可采用：

- 单独运行 `warm-flow-ui` 开发服务进行观察
- 或重新构建 UI 产物，再通过后端模块验证

第一版验证不要求一开始就改 UI 源码，应先基于现成功能建立认知。

### 7.2.4 调试收益

采用源码联调而非远端 jar 的主要收益：

1. 改框架逻辑后可立即验证
2. 可直接打断点进入引擎与 UI 后端代码
3. 可持续迭代 demo 场景，不受发布节奏约束
4. 更适合当前“边试边改边建立认知”的工作方式

### 7.2.5 第一版建议策略

第一版默认采用以下模式：

```text
框架源码仓库
  + 新增 warm-flow-demo-app
  + 本地数据库
  + 本地源码构建
  + 必要时修改框架源码
  -> 直接观察运行结果
```

### 7.3 主要风险

1. 条件表达式虽能保存，但运行时变量路径写错会导致流转失败
2. 表单配置界面支持的字段能力可能与预期不完全一致
3. 同一测试用户跑全链路时，可能掩盖部分真实权限问题
4. 宿主页只是调试壳，不代表真实 IRA 前端最终结构
5. 如果当前仓库局部实现与文档存在偏差，需要以实际运行结果为准修正认知

### 7.4 风险控制方式

1. 第一阶段只保留一个分支字段 `leaveDays`
2. 把表单配置界面纳入首轮验证，避免跳过关键认知环节
3. 先框架内闭环，再做宿主页，不把两类问题混排
4. 第二阶段只模拟用户切换，不引入真正权限系统
5. 所有结论都必须以实际操作与页面结果验证

### 7.5 数据准备与初始化约定

第一版默认采用：

- 数据库：MySQL
- 建表脚本：仓库自带 `sql/mysql/warm-flow-all.sql`
- 运行方式：本地 Spring Boot + 本地数据库

本次不优先使用 H2，避免把数据库兼容差异引入首轮认知验证。

#### 初始数据准备范围

至少需要准备：

- 1 个可用测试用户
- 1 套流程配置空间
- 3 份动态表单定义
- 1 份请假流程定义

#### 用户与权限策略

第一阶段：

- 默认只准备 1 个测试用户即可
- 不强依赖真实角色、部门、权限模型

第二阶段：

- 宿主页通过任务上下文切换模拟不同视角
- 如后续验证确实需要，再补最小多用户数据

#### 流程定义来源

第一版默认采用：

- 表单：通过系统自带表单配置界面手工创建
- 流程：通过系统自带流程设计器手工配置

不在第一版用 SQL 直接灌入流程定义，避免跳过关键认知过程。

### 7.6 首轮失败回退策略

如果第一阶段未能按预期跑通，不直接扩大修改范围，而是按下面顺序收敛问题：

1. 先判断失败点属于哪一层：
   - 表单配置界面
   - 节点绑定
   - `formData` 变量注入
   - 条件分支执行
   - `formCreate` 运行态渲染
2. 如果 `formCustom = Y` 的节点绑定链路不成立：
   - 先单独验证动态表单是否能被正常创建、保存、加载
3. 如果表单能渲染但条件分支不成立：
   - 将问题降级为“动态表单字段进入流程变量”的专项核查
4. 如果框架内闭环未跑通：
   - 暂不进入 iframe 宿主页阶段
5. 所有失败结论都需要记录实际页面表现、参数和日志，避免凭印象修正 spec

## 8. 后续实现顺序

实现阶段推荐按以下顺序推进：

1. 新增 `warm-flow-demo-app` 作为业务联调模块
2. 熟悉并验证系统自带表单配置界面
3. 通过配置界面建立三份动态表单定义
4. 建立请假流程定义与条件分支
5. 跑通同一测试用户下的框架内闭环
6. 建立最小宿主页并接入 `formCreate`
7. 补充“申请人 / 主管 / 经理”视角切换
8. 形成一份手动操作说明，记录如何配置表单、如何发起、如何切换分支、如何观察渲染结果

## 9. 验收结论

本设计的目标不是交付最终业务系统，而是交付一个**最小但真实的可视化认知样板**。

只要该样板能同时回答下面两个问题，本次设计就成立：

1. 使用 Warm-Flow 自带表单配置界面生成动态表单后，字段能否驱动条件分支并正常流转？
2. 将 `formCreate` 挂到外部宿主页后，动态表单与节点渲染的整体交互形态是什么样？

如果两点都能通过实际操作获得清晰答案，就说明该样板已经达到目的。
