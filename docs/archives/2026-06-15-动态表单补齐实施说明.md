# 动态表单分支变量校验实施说明

## 一、背景

当前 Warm-Flow 已经具备动态表单的基础能力，但在“流程定义阶段如何正确引用动态表单字段参与分支条件”这个点上，还没有形成稳定闭环。

现状问题不是“动态表单不能用”，而是：

- 动态表单字段定义属于**配置态数据**
- 分支条件表达式同样属于**配置态数据**
- 两者目前没有做联动校验

这会导致一个直接问题：

- 设计流程时，用户可以手动填写条件表达式
- 但系统不会检查表达式里引用的字段，是否真的存在于当前绑定的动态表单中

结果就是，流程定义可以保存成功，但运行到网关判断时才因为变量不存在而失败，错误暴露过晚，使用体验和排查成本都比较差。

## 二、目的

本次实施的目标，不是重构动态表单体系，也不是改造运行时变量模型，而是做一版**最小可落地、可上线、可显著降低配置错误**的收口方案。

本次目标明确为：

- 在流程定义阶段，对“自定义动态表单 + 分支条件表达式”的组合做基础校验
- 尽量把错误前移到“保存流程定义”时暴露
- 保持现有运行时机制不变

本次不追求：

- 自动下拉选择字段
- 运行时变量结构重构
- 业务表落库方案
- 表单字段映射体系

## 三、讨论提纲

### 1. 跳转条件怎么写

- 先不做字段下拉，继续手动输入
- 仅在动态表单场景下做校验
- 条件引用统一按 `formData.<field>` 约定

### 2. 校验什么时候做

- 在保存流程定义时做基础校验
- 只校验已保存、已绑定的动态表单字段
- 不替代运行时机制

### 3. 变量不存在会怎样

- 不会静默通过
- 运行时会在网关判断阶段失败
- 监听器不作为兜底路由机制

### 4. 表单数据存哪里

- 表单定义存 `flow_form.form_content`
- 运行时表单值存 `flow_instance.variable`
- 历史快照存 `flow_his_task.variable`

### 5. 方案边界是什么

- 不做自动下拉
- 不改运行时变量模型
- 不把流程变量当业务主存储

## 四、现状事实

### 1. 动态表单定义数据来源

动态表单设计器保存的是表单定义 JSON，而不是运行时数据。

当前保存结构为：

```json
{
  "rule": [...],
  "option": {...}
}
```

其中：

- `rule` 保存字段定义
- 字段 key 对应 `field`
- 字段类型对应 `type`

这意味着，**动态表单字段集合在定义阶段是可读取、可提取的**。

### 2. 动态表单运行时数据来源

运行时表单提交流程走的是框架自带的 `execute/handle`。

提交后的表单值不会单独落到业务表，而是进入流程变量：

```json
{
  "formData": {
    "name": "张三",
    "age": 18
  }
}
```

当前真实契约是：

- `FlowCons.FORM_DATA = "formData"`
- 动态表单提交值进入 `variable.formData`
- 不是平铺到根变量

因此，从引擎当前真实行为看，分支条件引用动态表单字段时，最稳妥的变量前缀应该是：

```text
formData.<field>
```

而不是：

```text
<field>
```

也不是：

```text
form.<field>
```

实现时应直接复用 `FlowCons.FORM_DATA` 常量，不要再硬编码 `"formData"`。

### 3. 条件判断失败时的行为

当前网关条件判断发生在运行时。

如果表达式引用了不存在的变量，系统不会自动兜底跳过，也不会自动转给下一个人，而是会在流转时抛出异常，导致本次流程推进失败。

也就是说：

- 不是“静默错”
- 也不是“自动卡死无提示”
- 而是“运行时失败，错误暴露过晚”

这正是本次要把错误前移到定义阶段的核心原因。

### 4. 动态表单绑定语义

需要明确：

- `formCustom = Y` 时，`formPath` 存的是动态表单主键 ID
- 后端根据该 ID 读取 `flow_form.form_content`
- 节点级 `formCustom/formPath` 优先，流程级作为兜底

### 5. 条件表达式的真实形态

当前条件配置至少分两类：

1. 比较型条件

```text
gt@@leaveDays|3
eq@@status|PASS
```

2. 表达式型条件

```text
default@@${formData.leaveDays > 3}
spel@@#{@bean.check(formData.leaveDays)}
snel@@#{formData.leaveDays > 3}
```

因此，本次校验不能只按 `${...}` 一种示例理解，而应按条件类型分别处理。

## 五、最终方案

### 1. 方案结论

本次采用“**手动输入 + 保存时校验**”方案。

即：

- 当前阶段不做字段自动下拉
- 条件表达式仍由用户手动输入
- 当流程/节点绑定的是动态表单时，保存流程定义时执行基础校验

### 2. 校验适用范围

仅对以下场景生效：

- `formCustom = Y`
- 且当前节点或流程绑定了动态表单
- 且分支条件表达式声明要引用动态表单字段

以下场景不纳入本次校验：

- `formCustom != Y`
- 页面路径模式
- 业务系统自己扩展的外部变量
- 非动态表单来源的表达式变量

### 3. 表单绑定优先级

当流程级和节点级同时存在表单绑定时，本次校验遵循与运行时一致的优先级：

- 节点级 `formCustom = Y` 且 `formPath` 有值时，以节点级为准
- 否则回退到流程级

也就是说，校验时不能只看流程级表单，必须按当前节点实际生效的绑定关系取字段集合。

### 4. 字段引用约定

本次统一约定：

```text
formData.<field>
```

作为动态表单字段在分支条件中的引用方式。

例如：

```text
${formData.leaveDays > 3}
```

或在自定义规则中识别到：

```text
formData.leaveDays
```

则认为它在引用动态表单字段 `leaveDays`。

### 5. 校验目标

保存时至少做两类基础校验：

1. 如果当前绑定的是动态表单，并且表达式中声明引用动态表单字段：
   - 必须满足 `formData.<field>` 约定
2. `<field>` 必须存在于当前绑定动态表单的已保存字段集中

如果不满足，则保存失败，并给出明确错误提示。

补充约束：

- `form.leaveDays` 这类明确错误前缀可直接判错
- `formData.leaveDays` 进入字段存在性校验
- 裸字段 `leaveDays` 仅在简单比较型条件下谨慎识别，复杂表达式不做强推断

### 6. 当前明确不做

本次不做：

- 不做字段自动补全或下拉
- 不做表达式语法完整解析器重构
- 不支持对所有复杂嵌套表达式做完美识别
- 不把 `formData` 自动投影为根变量
- 不新增 `form` 作为别名变量

## 七、后端改造

目标：在 `save-json` 链路加入定义态联动校验，形成真正有效的约束。

本轮实际把校验插在：

- `WarmFlowService.saveJson()`

不建议本轮下沉到 core 层，原因是：

- 本次校验依赖 `flow_form.form_content.rule`
- 这是动态表单/UI 插件层知识
- 不应把这类前端表单定义格式直接耦合进 `warm-flow-core`

#### A. 读取动态表单定义

在 `save-json` 对应后端链路中：

- 识别当前流程级/节点级是否绑定动态表单
- 按“节点级优先、流程级兜底”解析当前节点实际生效的 `formPath`
- 根据 `formPath` 读取对应动态表单定义
- 解析 `formContent.rule`
- 提取字段集合

#### B. 提取可用字段集合

从 `formContent.rule` 中提取：

- `field`
- 可选保留 `type`

本次校验最核心的是 `field`，`type` 仅作为后续扩展信息保留。

补充：

- 本轮实现已支持递归提取 `children`
- 用于覆盖动态表单中常见的嵌套字段场景

#### C. 校验分支条件表达式

对流程定义中的跳转条件做扫描。

需要关注：

- 网关条件
- 跳转线条件

本轮实际校验逻辑：

1. 对比较型条件：
   - 识别形如 `gt@@leaveDays|3`
   - 如果字段位命中动态表单字段，则要求改成 `formData.<field>` 约定
2. 对表达式型条件：
   - 提取其中显式出现的 `formData.<field>` 引用
   - 再校验 `<field>` 是否存在于字段集合中

补充边界：

- `form.leaveDays` 直接判错
- 复杂表达式里的裸字段不做强推断，避免误杀

如果发现明显错误，直接阻止保存。

#### D. 错误提示

错误提示需要明确、可读，避免只抛底层异常。

建议提示语义：

- 当前动态表单未包含字段：`xxx`
- 动态表单字段引用必须使用 `formData.<field>` 形式

## 十、验证清单

### 1. 动态表单字段存在，表达式引用正确

- 绑定动态表单
- 表达式使用 `formData.leaveDays`
- `leaveDays` 在表单定义中存在
- 保存成功

### 2. 动态表单字段不存在

- 绑定动态表单
- 表达式使用 `formData.notExistField`
- 表单定义中不存在该字段
- 保存失败，并给出明确提示

### 3. 使用错误前缀

- 绑定动态表单
- 表达式使用 `form.leaveDays`
- 保存失败，并提示应使用 `formData.<field>`

### 4. 裸字段场景

- 绑定动态表单
- 表达式使用裸字段 `leaveDays`
- 对简单比较型条件，可谨慎识别
- 对复杂表达式型条件，不做强推断式误杀

### 5. 非动态表单模式

- `formCustom = N`
- 表达式中不做这类动态表单字段校验
- 保存逻辑保持原样

### 6. 运行时回归

- 保存通过的动态表单流程定义
- 发起流程
- 办理节点提交动态表单
- 网关条件能正常依据 `formData` 进行判断

## 十一、风险与边界

### 1. 这不是完整表达式分析器

本次只做“够用”的校验，不追求覆盖所有复杂表达式写法。

对于特别复杂的自定义表达式：

- 可以先保证常见写法可识别
- 极端写法后续再扩展

### 2. 表单字段来源依赖已保存定义

只有已保存的动态表单，字段集合才稳定。

因此本次能力默认建立在：

- 已绑定的动态表单是可读取的
- 其 `formContent.rule` 是合法的

### 3. 租户与数据隔离

如果目标系统启用了租户隔离，那么后端读取 `flow_form` 时，必须遵守当前租户上下文。

### 4. 不替代业务数据建模

本次方案解决的是“流程定义校验”问题，不解决：

- 业务字段长期存储
- 统计检索
- 报表分析

这些仍然不适合长期只依赖 `variable.formData`。

## 十二、简短总结

本次方案的本质，不是扩展动态表单能力，而是给已经存在的动态表单运行链路补上一个“定义阶段校验收口”。

最终实施口径可以收成一句话：

- 当流程节点绑定的是动态表单时，分支条件允许手动引用 `formData.<field>`；保存流程定义时，系统校验该字段是否真实存在于当前绑定动态表单中。

这样做的收益是：

- 不改运行时
- 不扩大范围
- 能明显降低配置错误
- 足够支撑当前业务推进

### 4. 当前注意事项

本轮验证过程中发现，仓库根 `pom.xml` 首行如果带有前置空格，会导致 Maven 无法解析。

当前口径应明确为：

- 根 `pom.xml` 必须以 `<?xml ... ?>` 直接起始
- 否则整个项目 Maven 入口会直接失败，与业务改造本身无关
